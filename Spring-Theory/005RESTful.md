# REST API
Representational State Transfer의 약자로 자원을 이름으로 구분하여 해당 자우너의 상태를 주고받는 모든 것을 의미한다

- REST의 특징
1. 인터페이스의 일관성
- 리로스에서 수행할 수 있는 작업의 균일한 인터페이스를 정의한다
- get, post, put, delete와 같은 http method를 사용하여 구현된다

2. 무상태성
- 작업을 위한 상태 정보를 따로 저장하고 관리하지 않는다
- 세션 정보나 쿠키 정보를 별도로 저장하고 관리하지 않기 때문에 api 서버는 들어오는 요청만 단순히 처리한다

3. 캐싱
- reset은 http 웹표준을 그대로 사용하기 때문에 웹에서 사용하는 기존 인프라들을 그대로 활용할 수 있다
- http 프로토콜에서 사용하는 last-modified 태그나 e-tag를 이요하면 캐싱 구현이 가능하다

4. 서버 클라이언트 구조
- 영역을 클라이언트와 서버로 분리한다
- 유저 인터페이스와 서버를 분리하게 되면, 서버가 처리할 부분이 줄어들고 여러 플랫폼과 유저 인터페이스 형태로 확장하기 쉬워진다
    - 하나의 데이터를 웹, 앱, 그리고 여러 클라이언트에서 보여주어야흐는 요구사항에 rest api가 빛을 보게 된다

5. 계층형 구조
- reset 서버는 다중 게층을 구성될 수 있으며, 로드 밸런싱, 암호화, proxy 등의 계층을 추가해 구조상의 유연성을 둘 수 있다
- 해당 계층들은 클라이언트에 보이지 않는 상태로 유지된다

## REST의 장단점
### REST의 장점
- 프로토콜의 인프라를 그대로 사요하므로 인프라 구축 필요가 없다
- 프로토콜의 표준을 최대한 활용하여 여러 추가적인 장점을 함께 가져갈 수 있게 해준다
- http 표준 프로토콜에 따르는 모든 플랫폼에서 사용이 가능하다
- 서버와 클라이언트의 역할을 명확하게 분리한다

### REST의 단점
- 표준자체가 존재하지 않아 정의가 필요하다
- http method 형태가 제한적이다
- 브라우저로 테스트를 많이 하면 header 값 수정이 어려줘 테스트에 어려움을 느낄 수 있다

## REST API
**api**
-> 응용 프로그램에서 사용할 수 있도록 운영체제나 프로그래밍 언어가 제공하는 게술을 제어할 수 있게 만든 인터페이스이다
- api는 프로그램들이 서로 상호작용하는 것을 도와주는 매기앋
- 개발자들이 어플리케이션 코드를 작성하는 방법을 표준화 함으로써, 개발 과정이 간소화되는 장점이 있다.
- 어떠한 방식으로 정보를 요청해야 하는지, 해당 요청 시 어떤 형식의 특정 데이터를 전달 받을 수 있는지에 대해 정리한 일종의 규격을 의미한다

**REST API**
- 각 요청이 어떤 동작이아 정보를 위한 것인지를 그 요청 자체로 추론이 가능하다

rest api 설계 원칙
1. url는 자원을 나타내야 하며, 동사보다는 명사를, 대문자보다는 소문자를 사용하여야 한다
    - 일반적으로는 복수 명사로 표현해야한다
2. 고유한 객체를 가지고 올 때는 리소스 옆에 path 상에 id를 나타낸다
3. 마지막에 슬래시를 포함하지 않는다
4. 언더바 대신 하이픈을 사용한다
5. 조인된 결과를 url상에 나타내주고 싶을 경우에 사용한다
6. url에 파일 확장자를 포함하지 않는다
7. 행위를 포함하지 않는다 (행위를 나타내고 싶을 경우 url가 아닌 http 메소드를 통해서 나나탤 것)

## rest api 동작
- 서버는 클라이언트의 요청에 대한 응답으로 화면이 아닌 데이터를 전송한다
- 이때 사용하는 응답 데이터는 json 또는 xml등이다
- json
자바스크립트 객체 문법으로 구조화 된 데이터를 표현하기 위한 문자 기반의 표준 포맷이다
- 키와 값으로 구성된 정렬되지 않는 속성의 집합이다
- 쉼표로 나열된다
- 객체는 중괄호로 둘러쌓아 표현한다
- 배열은 대확ㄹ호로 둘러 쌓아 표현한다

## 요청과 응답과정
1. 요청
- 클라이언트는 http 요청을 통해 서버에 데이터를 보내고, 서버는 그에 대한 응답을 반환한다
- reset api는 응답 데이터로 json 타입의 데이터를 클라이언트에게 전송한다
- http 요청 메세지는 시작 라인, 헤더, 본문으로 구성되어 있다
**요청라인**
- http method : 요청의 의도를 담고 있는 메서드를 포함한다
- request target : 전송되는 목표 주소이다
- http version:버전에 따라 메세지 구조가 다를 수 있기 때문에 버전을 명시한다

**헤더**
- 추가 정보를 담고 있는 부분이다
- 호스트 : 요청하려는 서버 호스트 이름과 포트번호를 포함한다
- 유저 에이전트 : 클라이언트 프로그램 정보를 포함한다
-> 서버는 클라이언트 프로그램에 맞는 최적의 데이터를 보낼 수 있다
- authorization : 인증 토큰을 서버로 보낼 때 쓰인다
- 쿠키: 쿠키 값이 키-밸류 값으로 표현된다

**본문**
- 전송하는 데이터를 담고 있는 부분으로 전송하는 데이터가 없다면 바디 부분은 비어있다.

2. 응답
- 상태
버전: 서버가 사용하는 포토콜의 버전을 나타낸다
스테이터스 코드 : 서버가 클라이언트에게 보내는 세 자리 숫자로 요청의 성공,실패,리다이렉션등의 상태를 나타낸다
스테이터스 문자 : 상태코드에 대한 간단한 설명을 ㅗ살마이 읽을 수 있는 형태이다
- 헤더 
서버 : 웹서버의 조율를 나타낸다
로케이션 : 301, 302 코드일때만 보일 수 이쓴ㄴ 헤더로 서버의 응답이 다른 곳에 있다고 알려주며 해당 ㅜ이치를 저장한다
캐시 컨트롤 : 캐시 정책으로 캐시의 유효시간을 정한다
셋 쿠키 : 서버가 클라이언트에게 쿠키를 저장하라는 명령
