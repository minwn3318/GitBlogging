# framework
## 프레임 워크
- 프레임워크는 애플리케이션 개발 시 필수적인 코드, 알고리즘 등의 기능들을 위해 뼈대를 제공한다

## 라이브러리
- 라이브러리는 개발자가 개발하는 데 필요한 것들을 모다 둔 함수들의 나열로, 개발자가 필요로 할 때 호출하여 사용하는 방식을 취하고 있다.

## 프레임워크와 라이브러리 차이점
- 프레임워크 : 알아서 객체를 관리
- 라이브러리 : 개발자가 전체적인 사용 흐름을 만듦

스프링의 핵심
- 자바 언어 기반의 프레임워크
- 객체지향언어
- 즉, 스프링은 객체지향언어가 가진 강력한 특징을 살려내면서 애플리케이션 제작이 가능하게 해준다

스프림 프레임워크 특징
- pojo기반의 구성
    - 객체지향적인 원리에 충실하면서 환경과 기술에 종속되지 않고 필요에 따라 재활용될 수 있는 방식으로 설계된 오브젝트

    - 단순히 순수 자바 객체만을 사용해서 프로그래밍 코드를 작성한다

- 제어의 역행과 의존성 주입을 통한 객체 간의 구성
    - 제어의 역행
    - 코드의 최종 호출을 개발자가 하는 것이 아닌, 스프링 프레임워크 내부에서 이루어진다

**의존성 주입**
- 한 객체의 코드에서 다른 객체를 생성하거나 다른 객체의 메서드를 호출할 때, 의존성이 발생한다고 할 수 있다

한 객체가 다른객체의 메서드를 사용하기 위해서는 해당 객체의 존재를 알고 잇어야 한다

스프링 기반의 애플리케이션에서는 애플리케이션 코드에서 이루어지는 의존성 주입을 스프링이 대신해주며 클래스들 간의 결합을 느슨한 결합으로 만들어주어 유연성을 높여준다

**의존성에서 사용하는 것**
- 경량 컨테이너
스프링에서 자바 객체들을 관리하는 공간
- 빈 : 스프링 컨테이너가 관리하는 자바 객체
    - 스프링 컨테이너는 빈의 생성부터 소멸까지 개발자 대신 관리해준다
    - 스프링 컨테이너에 객체, 빈을 등록하는 이유는 객체간의 의존관계들을 스프링이 관리하도록 하기 위해서다

aop(관점 지향 프로그램)지원
- spring aop : 스프링 프레임 워크에서 제공하는 기능 중 하나로 관점 지향 프로그래밍을 지원하는 기술이다

spring aop는 로깅, 보안, 트랜잭션 등과 같은 공통적인 관심사를 모듈화 하여 코드 중복을 줄이고, 유지 보수성을 향상시키는 데 도움을 준다

트랜잭션 : 순서대로 잘 진행하게, 중간에 문제가 있어도 다시 첨부터하게 만드는 것

- 여러개 의 클래스에서 반복해서 사용하는 코드가 있다면 해당코드를 모듈화 하여 공통 관심사로 분리한다

solid
SRP : 단일 책임의 원칙
- 클래스는 단 한개의 책임을 가져야 한다
- 한 클래스가 수행할 수 있는 기능이 여러 개라면, 클래스 내부의 메서드끼리 강한 결합을 갖게 될 가능성이 커지게 된다
- 객체 지향 설계의 핵심은 응집도는 높게, 결합도는 낮게 프로그램을 설계하는 것, 이에 책임을 잘게 쪼개어 분리시킬 필요가 있다

ocp
- 개방 폐쇄 원칙
- 확장에는 열려있어야하고, 변경에는 닫혀있어야 한다
-기존의 코드를 변경하지 않고 기능을 수정하거나 추가할 수 있도록 설계해야한다
- ocp는 추상화와 상속 등을 통해 구현해 낼 수 잇다
- 자주 변화하는 부분을 추상화하여 기존의 코드를 수정하지 않고도 기능을 확장시킬 수 있게 설계하여 유연성과 재사용서, 유지보수성을 높이는 것이 핵심이다

-인터페이스가 업케스팅을 적극적으로 사용한 거였구나

lsp : 리스코프 치환 원칙
- 부모 객체와 자식 객체가 존재할 때, 부모 객체를 호출하는 동작에서 자식 객체가 부모 객체를 완전히 대체할 수 있다는 원칙이다
- 즉, 상속이 일어나면 하위 타입인 자식 객체는 상위 타입인 부모 객체의 특성을 가지며, 이를 바탕으로 확장시켜 나갈 수 있음을 의미한다

인터페이스분리원칙
- 객체는 자신의 호출하지 않는 메소드에 의존하지 않아야 한다
- 만약 인터페이스이 추상 메서드들을 범용적을 구현한다면, 그 인터페이스를 상속받은 클래스는 잣니이 사용핮 ㅣ않는 인터페이스라도 억지로 구현해야 하는 상황이 찾아올 수 있다

DIP : 의존관계 역전 원칙
- 상위 모듈이 하위 모듈을 사용할 때, 하위 모듈의 구체적인 구현체에 직접 의존하지말고, 이 둘을 연결하는 추상화에 의존하라는 원칙이다

- 하위 모듈을 가져다 사용하게 된다면, 해당 모듈에 변화가 있을 때마다 상위 모듈으리 코드를 자주 수정해야 됙 때문이다

bean container
빈 객체들을 설정 파일에서 읽어서 관리하고 사용자 대신 프레임워크 측면에서 객체들을 생서하거나 주입하는 컨테이너다

bean은 스프링이 생성하고 관리하는 자바 객체이다
빈들읠 생명 주기를 관리한다는 의미로 빈 팩토리라고 한다
스프링 컨테이너는 객체의 생명주기를관리하며 제어의 역전을 통해 스프링 컨테이너 안에서 관리되는 빈의 생명 주기를 관리한다
의존성 관리가 용이하다
기본적으로 스프링 컨테이너가 객체 인스턴스를 싱글톤 방식으로 관리한다
싱글톤

빈 팩토리
- 빈을 등록하고 객체 생성, 조회, 호출 등의 다양한 기능을 담당한다
- 빈 팩토리는 빈의 정의는 즉시 로딩해 오지만 빈이 사용되기 전까지는 인스턴스와 하지 않는다
- getBean 메소드를 통해 호출되면 빈 팩토리는 의존성 주입을 통해 빈을 인스턴스화 하고 빈의 특성을 설정하기 시작한다 즉, 해당 시점부터 빈의 생명이 시작된다

애플리케이션 컨텍스트
- 애플리케이션 컨텍스트를 상속해 확장한 컨테이너다
- 하는 역할은 빈 팩토리와 동일하며 추가로 스프링이 제공하는 여러 부가 기능을 제공한다
- 애플리케이션 컨텍스트는 빈 팩토리와는 달리 초기화 시점에 빈을 생성해 두기 대문에 빈이 필요할 대 즉시 사용할 수 있따

DI 의존성
- 의존성 주입은 의존성을 주입하는 것이다

결합도
- 의존성의 정도를 결합도라고 한다
    - 결합도가 높으면 강한 결합 이라고 표현한다
    - 결합도가 낮으면 느슨한 결합이라고 표현한다
- 일반적으로 뉴 연산자를 사용하여 객체를 생성할 때 강하게 결합하여
- 인터페이스와 같이 일반화 된 구성 요소에 의존하고 있을 때 느슨하게 결합하게 된다

생성자 호출 시점에 1번만 호출되는 것을 보장하기 때문에 이후 값을 변경하지 못하게 막을 수 있다
- 불변과 필수 의존관계에 사용된다
- 생성자가 1개만 존재하는 경우 @autowired를 생략해도 자동 주입된다
- 주입 받을 필드를 final로 선언할 수 있다

수정자 주입
- 필드의 값을 변경하는 수정자 메서드를 통해 의존관계를 주입하는 방법

aop
관점 지향 프로그래밍
- 핵심 관심사와 횡단 관심사에 대한 관점들로 프로그램을 분해하여 객체 지향이 추구하는 모듈을 효과적으로 지원하도록 하는 프로그래밍 기법을 의미한다
- 기능별로 모듈을 분리했음에도 불구하고 생기는 중복 코드의 단점을 해겨하고자 나온 방식으로, 공통 기능과 핵심 기능을 분리하여 필요할 때만 공통 긴으을 핵심 기능에 넣어주는 형식이다 

장점
- 유지보수성 향상
- 코드 재사용성 증가
- 테스트 용이성
- 에러 추적과 디버깅 용이성
- 시스템의 확장성 향상
- 명확한 책임분리

애스팩트
- 핵심 기능에서 분리한 부가 기능과 이를 어디에 적용할 지를 정의한 것이다
- 횡단관심사를 구현하는 코드를 캡슐화하는 역할이아
- 어드바이스와 포이트컷을 포함한다

어드바이스
- 애스팩트에서 정의되는 메소드이다
- 횡단관심사를 구현하는 실제 코드를담고 있다
- 비포 애프터 어라운드 등 다양한 종류가 존재한다

조인포인트
- 애스팩트가 적용될 수 있는 특정 지점이다

- aop 프레임워크에서 제공한다

포인트컷

- 조인포인트들을 필터링하여 애스팩트 적용 위치를 결정한다
- exectuion, within, this 등 다양한 표현식을 사용 가능하다
